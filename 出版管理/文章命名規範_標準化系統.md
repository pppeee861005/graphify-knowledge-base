## 🎯 核心原則

```
本地檔案名 ← 基於 Substack 發布標題 + 系列結構
```

## 🏷️ 檔名命名規範

### **格式定義**

```
[系列簡碼]_[集次]_[簡化標題]_[檔案類型].md

其中：
├─ [系列簡碼]     = 2-3 字母代碼
├─ [集次]         = S01E01 (Season 01, Episode 01) 或 Ep01
├─ [簡化標題]     = Substack 標題簡化版（不超過 30 字）
├─ [檔案類型]     = (可選) _draft/_plan/_v2 等版本標記
└─ .md            = Markdown 副檔名
```

### **系列簡碼設定**

| 系列名稱 | 簡碼 | 說明 |
|---------|------|------|
| 從毒癮到創造者 | ADDICT | Addiction → Creator |

| Software 3.0 | SW3 | Software 3.0 範式 |

## 📋 版本管理規範

### **檔案版本後綴**

```
無後綴       = 已發布版本（最終版）
_v2, _v3...  = 修訂版本（用於更新已發布內容）
_draft       = 草稿版（尚未發布）
_plan        = 計畫版（系列規劃、大綱）
_old         = 歷史版本（已過時但保留作參考）
```

### **版本管理範例**

```
AGENT_S01E01_蜂群Agent運算資源新範式.md
    ↓ (發布後若有修訂)
AGENT_S01E01_蜂群Agent運算資源新範式_v2.md
    ↓ (重大更新)
AGENT_S01E01_蜂群Agent運算資源新範式_v3.md

未發布階段：
ADDICT_S01E03_引力轉移創造比消費上癮_draft.md
    ↓ (完成初稿)
ADDICT_S01E03_引力轉移創造比消費上癮_draft_v2.md
    ↓ (準備發布)
ADDICT_S01E03_引力轉移創造比消費上癮.md (發布)
```

## 🔍 Substack 標題 → 本地檔名轉換指南

### **轉換步驟**

```
步驟1：提取系列 + 集次
  例：「第1篇｜蜂群Agent...」→ 系列 = Agent，集次 = E01

步驟2：查表取得系列簡碼
  例：Agent → AGENT

步驟3：簡化標題（保留核心信息）
  長標題：「解耦哲學：Anthropic 如何設計蜂群的靈魂」
  簡化：「解耦哲學蜂群靈魂」
  規則：
    ├─ 刪除冗餘詞（「如何」「為什麼」等疑問詞）
    ├─ 刪除子句分隔符（「：」「——」）
    ├─ 保留核心名詞 + 動詞 + 形容詞
    └─ 最多 30 字

步驟4：組合檔名
  例：AGENT_S01E02_解耦哲學蜂群靈魂.md
```

### **簡化標題的規則**

| 原標題 | 簡化版 | 說明 |
|--------|--------|------|
| 你不是沉迷，你是被設計成沉迷的 | 你不是沉迷是被設計的 | 縮短但保留核心對比 |
| 解耦哲學：Anthropic如何設計蜂群的靈魂 | 解耦哲學蜂群靈魂 | 刪除冒號和「如何」 |
| 一張照片，如何摧毀一個產業的護城河 | 一張照片摧毀護城河 | 刪除冗餘詞 |
| Software 3.0的散戶基礎設施 | 散戶基礎設施 | 核心詞組 |
| 引力轉移：為什麼創造比消費更上癮 | 引力轉移創造比消費上癮 | 保留核心論述 |

## ✅ 應用到現有檔案的計畫

### **第一階段：統計與映射（1小時）**
```
1. 列出所有現存的本地檔案名
2. 對應到 Substack 發布標題
3. 創建映射表
```

### **第二階段：重命名現有檔案（2小時）**
```
1. 按新規範重命名現有檔案
2. 移到規範目錄結構
3. 更新內部引用（若有）
```

### **第三階段：未來維護（持續）**
```
1. 所有新檔案按此規範創建
2. 每次發布時更新映射表
3. 定期檢查目錄結構一致性
```

## 📌 重要提醒

1. **映射表優先於檔名**
   - 本地檔名可能改變，但映射表必須保持更新
   - 映射表是唯一的真實來源

2. **簡碼的一致性**
   - 同一系列必須使用同一簡碼
   - 不允許「AGENT」和「SWARM」混用

3. **版本號遞進**
   - 版本號從無後綴 → v2 → v3...
   - 不允許跳號
