## 🚨 應急預案

### 情景1：發布週期延遲（突發事件導致無法按時發布）
**觸發條件**：距離預計發布日期 <24小時 仍未完稿
**應對方案**：
1. 立即推遲發布至隔日 +12:00（避免倉促發布）
2. 在Substack上發布「編者按」說明延遲，並預告確切時間
3. 補償方案：發布延遲篇章時，同時發布一篇「碎片思考」（500字輕量級內容）

### 情景2：績效下滑（某篇文章打開率<20%）
**觸發條件**：發布3天後，打開率仍<20%
**應對方案**：
1. 檢查標題是否有理解偏差（對標高績效篇章）
2. 在該篇文章下發布「編者按」，用新的角度重新引導
3. 在下篇發布時提及該篇（二次引流）
4. 修訂該篇內容，在下月再發「優化版」

### 情景3：訂閱增速突降（單週增速<15人）
**觸發條件**：連續2週增速<15人
**應對方案**：
1. 緊急分析：檢查該週發布內容的共性（質量、選題、時機）
2. 加強社交推廣：在Twitter/LinkedIn主動分享最近高績效文章
3. 發起粉絲互動：提前舉辦直播、在評論區增加互動
4. 內容調整：下週優先發布「高確定性」話題（Agent、Software 3.0主線）

## 📝 發布前檢查清單（每篇文章發布前務必檢查）

```
□ 標題檢查
  ✓ 長度是否≤12字（避免失去衝擊力）
  ✓ 是否包含具體數字或對比詞
  ✓ 開頭是否有問題式或驚嘆式觸發
  ✓ OCR準確性（與Word/原始文檔對比）

□ 內容檢查
  ✓ 是否有系統性的邏輯框架（而非單點論述）
  ✓ 是否有具體案例或數據支持
  ✓ 全文800–1,500字（不超過2,000字）
  ✓ 結尾是否有「後續預告」（引導訂閱）

□ 格式檢查
  ✓ 標題層級是否正確（H1為標題，H2為段落）
  ✓ 列表、引用、代碼塊是否正常渲染
  ✓ 超連結是否有效（若有外部引用）

□ 發布檢查
  ✓ 發布時間是否為預定時間（避免太早/太晚）
  ✓ 是否在發布後15分鐘內檢查渲染效果
  ✓ 是否在Twitter/LinkedIn同步預告

□ 發布後檢查（發布3小時內）
  ✓ 瀏覽數是否開始增長（正常>10次/小時）
  ✓ 是否有讀者評論/問題需要回應
  ✓ 是否需要進行標題優化（若打開率低於預期）
```
