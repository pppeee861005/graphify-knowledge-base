# search_files vs graphify query

## search_files

**目的**：在檔案系統中直接搜尋文字內容（或檔名）。

- **工作方式**：使用 ripgrep 走硬碟，逐個檔案掃描符合正則或字串的行，不需要任何知識圖或 LLM。
- **優點**：速度快、不用消耗 API 配額，適合找出「原始文字出現位置」。
- **限制**：只能找到字面相符的片段；若內容用同義詞、不同說法或被拆分成不同節點，就會漏掉。
- **範例輸出**：

  ```
  ./ai-commander-hq/🧠學習備忘錄/多智能體工程課程表.md|37|起頭| 日期 | 課程 | 狀態 |
  ./ai-commander-hq/🧠學習備忘錄/多智能體工程課程表.md|376|起頭|- [ ] 5/14: 錯誤處理機制
  ```

---

## graphify query

**目的**：在已建構的知識圖（knowledge graph）中進行語意檢索，找出與問題相關的節點及其鄰近資訊。

- **工作方式**：先把原始檔案（Markdown、程式碼等）經過 AST + 語意抽取（使用 LLM）轉成節點與邊；之後只在記憶體中的圖上做 BFS/DFS（免費，不再呼叫 LLM）。
- **優點**：能捕捉同義詞、上下文概念，例如「錯誤處理」可能對應到節點 `錯誤恢復機制`、`容錯策略`、`重試機制` 等，即使原文沒出現這 exact 字眼也能找到。
- **限制**：需要先建立圖（`graphify add` 或 `extract`），首次建會消耗 LLM 配額；之後查詢才是免費。
- **範例輸出**（假設圖中已有相關節點）：

  ```
  錯誤處理機制 → 錯誤恢復率 > 80%
  錯誤處理機制 → 設計重試、回退、超時策略
  ```

---

## 小結

| 項目 | search_files | graphify query |
|------|-------------|----------------|
| **消耗資源** | 零（純 I/O） | 首次建圖需 LLM，之後免費 |
| **搜尋類型** | 字面比對 | 語意 / 圖結構鄰近 |
| **適用情境** | 快速定位原始文字、除錯、grep-like 需求 | 想了解概念之間的關係、探索知識結構、做「類似搜尋」 |
| **需要前置工作** | 無 | 需要先把文件匯入知識圖（`graphify add ...`） |
| **範例用法** | `search_files --pattern "錯誤處理" --target content --file_glob "*.md"` | `graphify query "錯誤處理" --graph /path/to/graph.json` |

---

## 練習建議

在今天的練習中，你可以：

1. 先用 `search_files` 確認哪些檔案真的提到「錯誤處理」這四個字
2. 再把那些檔案（或整個知識庫）匯入 Graphify
3. 用 `graphify query` 看看是不是還能發現更多「錯誤恢復」「容錯」等相關概念

這正是從純文字檢索升級到知識圖問答的典型流程。祝你玩得開心！ 🚀
