## 1969 年的一個問題

1969 年，Bell Labs 的 Ken Thompson 在設計 Unix 時面對一個看似不可能的問題——

**如何寫一個作業系統，讓它能運行一個還沒被想出來的程式？**

那個程式不存在。沒有人知道未來十年會有什麼樣的軟體被寫出來。但作業系統必須現在就決定：它要怎麼讀檔案、怎麼分配記憶體、怎麼讓不同的程式互相通訊。

Thompson 的答案，是後來改變整個計算機世界的一句話——**「為尚未被想到的程式而設計」**。

他發明了一套接口（open、read、write、close），這些接口不關心你的程式長什麼樣、用什麼語言寫、想做什麼。它們只規定一件事——**程式和系統之間，用什麼方式對話**。

於是 1969 年寫的 read() 指令，到 2026 年仍然能用。從磁帶到磁碟，從機械硬碟到 SSD，再到 NVMe——底層儲存技術換了四代，但 read() 沒變過。**因為設計的對象不是儲存技術，而是「程式如何讀取資料」的關係。**

這就是「解耦」的真正含義。它不是工程技巧，是一種**為未來而設計的思維方式**。

2026 年，Anthropic 在設計蜂群時，面對的是同一個問題——

**如何讓一個 Agent 系統，能執行一個還沒被發明的 LLM？**

---

## 第 1 幕：三層分離

Anthropic 把蜂群拆成三個徹底分離的層次。理解這個分離，是理解蜂群一切可能性的鑰匙。

**大腦（Brain）**：只負責推理與決策。它接收任務，吐出判斷。它不知道自己是 Claude Sonnet 還是 Opus，也不知道執行環境是 AWS 還是地端。它只做一件事——思考。

**框架（Harness）**：負責「讓大腦能工作」的所有雜事。呼叫哪個模型 API、把工具的回應整理成大腦看得懂的格式、處理超時與重試、路由錯誤——這些都在 Harness 裡。它是大腦和外部世界之間的翻譯層。

**執行（Session）**：持久層。記錄所有事件，管理所有狀態，協調所有單蜂。Session 不思考，但它是整個蜂群唯一的記憶。任何單蜂死了，Session 還在。

這聽起來像我們熟悉的三層架構（MVC、Clean Architecture 那些）。但有一個本質區別——

**傳統三層架構，是「同一個系統」的內部切分**。Controller、Service、Repository 雖然分層，但它們是同一個應用程式的不同部分，會一起發布、一起升級、一起退役。

**Anthropic 的三層分離，是「三個獨立物種」的協作**。大腦由 Anthropic 升級，框架由你升級，執行層永遠存在。它們之間沒有依賴，只有約定。

這個區別決定了一切。

當 Anthropic 發布新模型時，大腦升級——你的框架不需要改。
當你想換工具、改路由、加新模態時，框架升級——大腦的判斷邏輯不需要改。
當你的業務邏輯、記憶結構、執行軌跡累積成資產時，Session 沉澱——大腦和框架怎麼換都不影響。

**三層各自進化，三層各自永生。**

這就是蜂群「拒絕老化」的工程基礎。可適應性不是寫在某個模組裡的功能，而是這套分離架構的副產品。

## 第 2 幕：Harness 會過時，Session 永存

但這套分離架構，並不是說「所有層都同樣重要」。

Anthropic 工程文件裡有一個非常微妙、容易被忽略的判斷——

**「Harness 的假設會過時，Session 的接口永存。」**

這句話需要解碼。

Harness 是針對「當下這代模型」設計的。Claude Sonnet 4.5 有「上下文焦慮」——當上下文接近極限時，回答品質會下降。所以 Sonnet 4.5 的 Harness 裡，要寫一套上下文管理策略：什麼時候壓縮歷史、什麼時候摘要、什麼時候切換新會話。

這些策略寫進 Harness 後，整個系統表現很好。

然後 Opus 4.5 發布了。**Opus 4.5 沒有上下文焦慮**。它的長上下文處理能力天然就強，原本 Sonnet 需要的那套策略，現在不僅多餘，甚至會讓 Opus 表現變差（因為策略強行壓縮了 Opus 原本能用的資訊）。

Sonnet 時代的 Harness 假設——「LLM 在長上下文下會焦慮」——**過時了**。

但這完全沒關係。因為 Harness 本來就被設計成可丟棄的。你寫一套新的 Harness 配合 Opus 4.5，舊的丟掉，整個系統繼續跑。**只有 Harness 在更新，大腦和 Session 都不動。**

這就是「Harness 會過時」的真實含義——它本來就該過時。每一代模型的 Harness，都是針對那一代模型的特性而寫的「適配層」，它的生命週期等於那一代模型的生命週期。

那麼，永存的是什麼？

**是 Session 暴露給外界的接口。**

Session 對外只承諾一件事：**任何單蜂，都可以透過一個統一的接口，讀取過去發生過的所有事件**。

無論單蜂是 Sonnet 還是 Opus，無論它要做什麼任務，無論它從哪個 Harness 啟動——它都能透過同一個接口拿到所有歷史狀態。

這個接口可以再具體一點：它叫 **getEvents()**。

## 第 3 幕：getEvents() 與事件日誌

getEvents() 是整個蜂群架構的支點。

要理解它的威力，要先看傳統 Agent 是怎麼做的。

**傳統 Agent 用的是「記憶體」概念**。每個 Agent 有自己的對話歷史、自己的 context、自己的 state。當 Agent A 完成任務交接給 Agent B 時，A 要把自己的記憶整理成 prompt 傳給 B。B 拿到 prompt 後，重新建立自己的記憶。

這種方式有兩個致命問題：

第一，**記憶是私有的**。A 的記憶 B 看不到，除非 A 主動交出。這意味著真正的並行很難實現——10 個 Agent 同時工作，它們的記憶各自孤立，最後彙總時要做大量的協調與去重。

第二，**記憶會失真**。每一次「整理成 prompt」傳遞，都是一次有損壓縮。任務鏈拉長後，最後一個 Agent 看到的世界已經和最初發生的世界完全不同。

Anthropic 的 Session 用的是另一個思路——**事件日誌（Event Log）**。

Session 不儲存「誰的記憶」，它只儲存「發生了什麼事」。每一個事件——某個單蜂啟動了、某個工具被呼叫了、某個結果回來了、某個錯誤發生了——都以事件的形式按時間順序寫進 Session。

任何單蜂在啟動時，呼叫 getEvents()，就能拿到截至此刻整個蜂群發生過的所有事件。

這個設計的革命性，在三件事上同時體現：

**第一，並行第一次成為可能**。10 個單蜂同時工作，它們不需要互相打聽「你做到哪了」——大家都讀同一份事件日誌。沒有同步成本，沒有交接成本。

**第二，新蜂可以無縫加入或替換死掉的蜂**。某隻單蜂掛了？Session 在毫秒級啟動一隻新蜂，新蜂讀完 getEvents() 就立刻接手——它擁有死蜂擁有過的所有上下文，沒有任何損失。

**第三，未來的蜂也能讀懂今天的事件**。因為事件日誌的格式是約定，不是實現。今天用 Sonnet 4.5 寫進去的事件，明年 Opus 5 的單蜂讀進來，一樣能用。**事件不會老化，因為它記錄的是「發生過什麼」，這是物理事實。**

這就是為什麼 read() 從磁帶到 SSD 都不需要改——因為 read() 描述的不是「儲存技術」，而是「讀取的關係」。

getEvents() 也一樣。它描述的不是「LLM 怎麼推理」，而是「單蜂如何取得歷史」。LLM 換了一代又一代，這個關係不變。

**Anthropic 在 Session 層做的事，是把「歷史」變成一種與模型無關的物理事實。**

這是整個蜂群架構最深的一層哲學。

## 第 4 幕：實戰一瞥

理論講到這裡夠了。讓我用最短的篇幅，給你一個真實案例的高潮點。

Netflix 在 2025 年導入多 Agent 編排處理基礎設施故障。過去當系統出問題時，工程師要看日誌、查監控、串連幾個 dashboard、寫 query、跑診斷——平均處理時間以「小時」計。

導入蜂群後：Lead Agent 接到告警 → 分發 10 隻並行子蜂（一隻看日誌、一隻看網路、一隻看資料庫、一隻看部署紀錄……）→ 所有子蜂共享 Session 的 getEvents()，互相不阻塞 → 結果在分鐘級彙總給工程師。

**從小時到分鐘——縮短的不是工程師的工作，而是「等待答案」的時間。**

這個案例之所以重要，不在於它證明蜂群「快」。而在於它證明：**當你用對了 Session 架構，並行不再是奢侈品，而是預設值**。

## 收尾：寫給設計師的一個提問

第 1 篇定義了蜂群「是什麼」。第 2 篇論證了它「為什麼不會老化」。這一篇——我想留給你一個提問。

**你今天寫的這段 Agent 邏輯，10 年後還能用嗎？**

這不是一個刁難。這是一個非常實際的設計檢查。

如果你的邏輯寫在 prompt 裡，那它的壽命 = 這代模型的壽命。下一代模型出來，你要重寫 prompt。

如果你的邏輯寫在 Harness 裡，那它的壽命 = 這代模型的特性週期。模型換代，你要重寫 Harness。

如果你的邏輯寫在 Session 的接口設計裡——**你描述的是「單蜂如何協作」、「歷史如何被讀取」、「任務如何被分解」這些不會隨模型而變的關係**——那它的壽命，等於蜂群這個物種的壽命。

Ken Thompson 在 1969 年沒有預測到 SSD。但他寫的 read()，今天的 SSD 還在用。

你今天設計的 Session 接口，可能會在 2036 年的某個 AGI 系統裡，仍然被一隻單蜂呼叫。

**這就是「為尚未被想到的程式而設計」的真正力量。**

它不是預測未來。它是用一種特別的架構姿態，**讓未來無論變成什麼樣，你的設計都還站在那裡**。
