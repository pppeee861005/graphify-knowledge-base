### 上午工作流程

**具體行動 1：API 認證方式研究**
- 嘗試 6 種不同的 HTTP 認證方式，全部失敗（401）
  - Bearer + Access Key ❌
  - Bearer + Secret Key ❌
  - Payload 中包含密鑰 ❌
  - AWS Signature V4 ❌
  - Basic Auth ❌
  - 自定義 Header ❌

- **突破點**：GitHub 搜尋發現正確方式
  - 需要用 AccessKey + SecretKey 生成 JWT Token
  - 使用 HMAC SHA256 算法
  - Token payload 必須包含：`iss`、`iat`、`nbf`、`exp`

**具體行動 2：JWT Token 生成實現**
- Python 實現（使用 PyJWT 2.10.1）：
  ```python
  import jwt
  import time

  def make_kling_token(access_key: str, secret_key: str) -> str:
      now = int(time.time())
      payload = {
          "iss": access_key,      # issuer
          "iat": now,             # issued at
          "nbf": now,             # not before
          "exp": now + 300        # expires in 5 minutes
      }
      return jwt.encode(payload, secret_key, algorithm="HS256")
  ```

- 關鍵發現：
  - `nbf` (not before) 欄位是必需的，否則返回 401
  - Token 有效期 5 分鐘（需要定期更新）
  - Authorization header 格式：`Bearer {token}`

## 💡 重要心得 / 發現

| 發現 | 說明 |
|------|------|
| **JWT 認證的關鍵細節** | Kling API 使用 HS256 JWT，不是常見的 Bearer + Key 方式。Token 必須包含 `nbf` 欄位，5 分鐘過期 |
| **測試驅動的發現** | 6 種 HTTP 認證方式失敗後才找到正確方向，說明 API 文檔不夠清晰，需要通過代碼示例反推 |
| **.env 格式的重要性** | 原始 .env 格式（多行無法解析）導致初期認證失敗，標準化環境變數格式後問題解決 |
| **API 端點的一致性** | 不同的 API 使用不同的基礎 URL 前綴（api.klingai.com vs api.kling.ai），容易混淆 |
| **帳號額度限制** | 現有 Access Key 已無可用額度，需要充值或更換帳號才能實際生成視頻 |
| **任務查詢端點缺失** | 嘗試了 16 個可能的任務查詢端點，全部返回 404，說明：(1) 文檔不全 (2) 需要特殊權限 (3) 可能需要在 Web 界面查看 |
| **異步 HTTP 客戶端的必要性** | 使用 httpx + asyncio 能更優雅地處理 API 調用和超時 |
| **工具模塊化設計** | 將認證、生成、查詢分離為獨立腳本，便於未來擴展和維護 |
